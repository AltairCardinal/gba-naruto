# Battle Configuration Format

## Status

**Substantially complete.** WRAM battle data layout identified via code analysis. ROM-based battle config tables located and documented. Runtime verification blocked by headless navigation limitation (no key input support in mGBA debugger). Both ROM patches and WRAM cheat patches are generated by `import_battle_config.py`.

## Key ROM Data Tables

### Unit ID Mapping Table

- **ARM address**: `0x0853F298`
- **File offset**: `0x53F298`
- **Size**: 128 bytes (u16[64])
- **Purpose**: Maps internal slot index → character ID used by the battle system

| Index Range | Values | Description |
|---|---|---|
| 0-9 | 0x00-0x09 | Player units (Naruto=0, Sasuke=1, Sakura=2, Sai=3, Kakashi=4, Shikamaru=5, etc.) |
| 10-29 | 0x19-0x2E | Special/enemy units |
| 30-42 | 0x0A-0x18 | Additional character units (Zabuza=10, Haku=11, etc.) |
| 43-55 | signed i16 | Movement offsets for 8-directional unit movement |
| 62-63 | 0x0C3D, 0x0808 | ROM data index marker |

### Battle Scenario Config Table

- **ARM address**: `0x0853D910`
- **File offset**: `0x53D910`
- **Entry size**: 16 bytes
- **Total entries**: 16 (8 valid, 8 padding)

**Entry format:**
| Offset | Type | Name | Description |
|---|---|---|---|
| +0x00 | u16 | tiles_x | Battle arena width in tiles |
| +0x02 | u16 | tiles_y | Battle arena height in tiles |
| +0x04 | u32 | ptr1 | ROM pointer to compressed tilemap data (12-byte header + raw tile IDs) |
| +0x08 | u32 | ptr2 | ROM pointer to LZ77-compressed palette/attribute data (384 bytes decompressed) |
| +0x0C | u16 | flag | Scenario flags |
| +0x0E | u16 | extra | Extra data / padding |

**Valid scenarios:**
| Entry | Dimensions | ptr1 (file) | ptr2 (file) |
|---|---|---|---|
| 0 | 92×64 | 0x0B9F80 | 0x0BD608 |
| 2 | 60×30 | 0x0BF318 | 0x0C166C |
| 4 | 36×36 | 0x0C1CF8 | 0x0C416C |
| 6 | 36×40 | 0x0C4880 | 0x0C5F24 |
| 8 | 36×40 | 0x0C6450 | 0x0C8454 |
| 10 | 36×46 | 0x0C8AF4 | 0x0CAA94 |
| 12 | 60×32 | 0x0CB18C | 0x0CCB54 |
| 14 | 60×32 | 0x0CD260 | 0x0D0054 |

### ptr1 Data Format (Tilemap)

Each ptr1 points to a block with a 12-byte header followed by tile data:

| Offset | Type | Description |
|---|---|---|
| +0x00 | u16 | Constant 0x0010 (16) |
| +0x02 | u16 | Stride/width parameter |
| +0x04 | u16 | Height parameter |
| +0x06 | u16 | Constant 0xF000 |
| +0x08 | u16 | Constant 0x9001 |
| +0x0A | u16 | Subtype/palette identifier |
| +0x0C | varies | Raw tile data (u16 per tile: lower 10 bits = tile ID, bits 10-15 = flip/palette) |

### ptr2 Data Format (LZ77 Compressed)

Each ptr2 points to LZ77-compressed data:
- **Header byte**: 0x10 (GBA LZ77 type)
- **Decompressed size**: 384 bytes (constant across all entries)
- **Content**: Palette or attribute data (BGR555 u16 values dominate)

Entries 2, 4, 6, 8 share identical compressed data, indicating a common palette/attribute set.

### Battle State Machine Function Pointer Table

- **ARM address**: `0x0853F1C0`
- **File offset**: `0x53F1C0`
- **Size**: 240 bytes (u32[60])
- **Content**: 60 function pointers to battle system handlers in ROM bank 0x0812Fxxx

## WRAM Battle Data Addresses

Located through cross-referencing ROM code (Thumb instruction literal pools).

| WRAM Address | Size | Code Refs | Description |
|---|---|---|---|
| `0x0201BE2A` | variable | 74 | Unit count + team count |
| `0x02021E2C` | 234*N | — | Unit lookup table base |
| `0x02022E30` | 0x127C (4732) | 50 | Battle state structure |
| `0x020240C0` | 0x17C4 (6084) | 140 | Main battle data |
| `0x02024294` | 234*25 | 2 | Unit array |
| `0x02026804` | 0x0008 (8) | 88 | Battle control flags |

### Memory Allocation Table

Located at ROM `0x0853D848` (file `0x53D848`). Contains `{u32 wram_addr, u32 size}` pairs.

### Unit Structure

- **Size**: 236 bytes (0xEC) per unit
- **Max units**: ~25 (6084 / 236)
- **Loop bound**: Code at `0x06D46C` iterates 12 units

Known field offsets:
| Offset | Type | Evidence |
|---|---|---|
| +0x00 | u8 | Existence flag (==0 means empty) |
| +0xBC (188) | u8 | Copy source for offset 0x00 |
| +0xD0 (208) | u32 | Non-zero check |
| +0xD4 (212) | u8 | Index value |

## Runtime Verification

**Blocked**: mGBA headless debugger does not support key input injection. The game cannot be navigated to battle state without user input. Previous attempts:
1. 10000-frame advance: only 5 non-zero words in battle WRAM (general game state, not battle data)
2. WRAM force-write of battle flags: did not trigger battle state
3. Lua script (mgba_battle_walk.lua): system mGBA lacks `--script` support

**Recommended**: Use mGBA Qt with Xvfb and Lua scripting, or save a savestate during battle for headless analysis.

## Investigation History

1. ASCII string search found battle state machine event labels
2. Static analysis: 38 late-ROM table candidates are visual/resource descriptor tables
3. WRAM analysis identified battle addresses through code literal pool cross-referencing
4. Code analysis identified unit struct size (236 bytes), battle data array layout
5. Watchpoint test: `0x020240C0` first write is BIOS zeroing WRAM during boot
6. Battle scenario config table located at `0x0853D910` with 8 valid entries
7. Unit ID mapping table confirmed at `0x0853F298` (u16[64])
8. ptr2 data confirmed as LZ77-compressed (decompressed size = 384 bytes)
9. 10000-frame headless advance: battle WRAM remains zeroed (game never reaches battle)
